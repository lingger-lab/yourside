# 곁에(yourside) — 제품 요구사항 문서 (PRD)

> **문서 버전**: 1.0 (마스터 문서 v3.1 기반)
> **대상 독자**: Claude Code, 개발팀(김동현 대표)
> **목적**: MVP를 무엇을, 누구를 위해, 어디까지 만들지 정의한다.

---

## 0. 한 줄 정의

부울경(부산·울산·경남) 시니어·청년의 경험을 검증해, 일손이 필요한 지역 사장님과 **사람이 직접 매칭**하는 로컬 인력매칭 플랫폼. AI 도구로 양측 모두에게 월 100만원+ 가치를 지향한다.

---

## 1. ⚠️ MVP 범위 경계 (가장 중요 — 과설계 금지)

**Claude Code는 이 경계를 반드시 지킨다. 아래 "포함"만 만들고, "제외"는 만들지 않는다.**

### 1.1 MVP 포함 (이번에 만든다)
- 소셜 로그인 (구글·카카오·네이버) — Supabase Auth
- 최소 정보 수집 (이메일·연락처 필수, 나머지 점진 수집)
- 사장님: 일 맡기기 → 의뢰 현황 → 견적 승인 → 검수·정산
- 파트너: 등록 → 의뢰 수락 → 작업 현황 → 정산·평가
- 관리자(3인): 통합 대시보드(매칭/진행/정산/문의) + 곁에 검토 입력
- 에스크로 결제 (토스페이먼츠)
- 간단 FAQ 챗봇 (14개 정적 응답) + 카카오 채널 연결
- AI 매칭 = **단순 키워드 매칭** (관리자 보조도구, 고객 비노출)
- 집출 서비스를 의뢰 유형으로 통합 (동일 매칭 흐름)
- **공통 5단계 워크플로우 — 수준 1 최소(신규)**: 파트너 작업현황에 공통 5단계(수집→구조화→생성→검증→납품) 체크리스트 + 진행 추적. 도메인 구분 없는 공통 뼈대 1개. 박이사·김동현 대표에게 "워크플로우가 거래에 붙어 작동함"을 보여주는 개념 증명. (상세: 곁에_파트너워크플로우_기초설계서.md)

### 1.2 MVP 제외 (지금 만들지 않는다 — 과설계 방지)
- ❌ RAG·고급 챗봇 — **새로 만들지 않음**. 김동현 대표 공용 모듈 연동 (1.5 참조)
- ❌ AI 임베딩·벡터 검색 매칭 (단순 키워드로 충분)
- ❌ 도구 구독 결제 시스템 (MVP는 도구 무료)
- ❌ 매칭 SaaS·화이트라벨
- ❌ 자동 정산 (관리자가 수동 버튼으로 실행)
- ❌ 모바일 네이티브 앱 (반응형 웹으로 충분)
- ❌ 도메인별 워크플로우(설정 5요소) → Phase 2 (MVP는 공통 뼈대 1개만)
- ❌ 워크플로우 단계별 AI 도구 임베드(수준 2) → Phase 2
- ❌ 워크플로우 데이터 학습 루프(수준 3) → Phase 3

### 1.3 단계적 DB 구현
- 극초기: `client`, `partner`, `request`, `deal` 4테이블부터.
- `matching`, `settlement`은 거래량 적을 때 관리자가 수기(노션/엑셀) 운영 가능 → 거래 쌓이면 DB 승격.
- (DB 상세는 `03_DB.md` 참조)

### 1.4 첫 마일스톤
"첫 50건 손매칭" — 사람이 직접 매칭하며 데이터를 쌓는 검증 단계. 이 데이터가 향후 AI 매칭 학습 교재가 된다.

### 1.5 RAG·챗봇 연동 (외부 모듈)
- RAG·챗봇은 **새로 만들지 않는다**. 엔터랩스가 선거솔루션(CampOne)에서 검증했고 김동현 대표가 진행 중인 **공용 RAG·챗봇 모듈(김동현 대표 관리)**을 연동한다.
- **MVP 1단계**: 곁에 자체 간단 FAQ 봇(14개) — 모듈을 기다리지 않고 출시.
- **MVP 2단계**: 모듈 준비 시 API로 연동, FAQ 봇 교체.
- 곁에는 **연동 인터페이스(API 계약)만 정의**하고 내부 구현은 모듈에 위임. (`04_TRD.md` 참조)

## 2. 사용자 (3주체)

| 주체 | 정체 | 핵심 니즈 | 호칭 |
|---|---|---|---|
| 사장님 | 수요자(사업자) | 검증된 사람에게 일을 맡기고 싶다. 떼이거나 부실할까 불안 | "사장님" |
| 파트너 | 공급자(시니어·청년) | 경험으로 일을 받고 싶다. 작업료 떼일까·일 없을까 불안 | "파트너" |
| 관리자 | 곁에 3인(박이사·Brad·김동현) | 의뢰↔파트너를 직접 검토·연결, 정산·검토 관리 | "곁에 검토진" |

추가 행위자:
- **곁에 매니저**: 여성 톤·정중한 비서실 직원 성격의 챗봇. 안내·연결·응대 담당(조언·판단은 사람이).

## 3. 핵심 사용자 흐름 (거래 7단계)

```
① 수요 등록 (사장님이 일 맡기기)
② 3인 검토·선별 (관리자가 AI 추천 후보 보고 파트너 직접 선택)
③ 견적 제시·승인 (곁에가 표준 단가표로 견적 책정 → 파트너 수락 → 사장님 승인)
④ 에스크로 선입금 (사장님이 작업료+매칭피 선입금, 토스페이먼츠)
⑤ 작업 수행 (파트너가 작업 후 결과물 제출)
⑥ 검수·정산 (사장님 검수 완료 → 관리자가 정산 실행 → 파트너에 작업료 전달)
⑦ 양방향 평가 (곁에 3인 검토진이 별점·의견 기록)
```

> 견적 권한 모델: **곁에가 표준 단가표로 책정, 파트너는 수락/거절권만** (모델 B+α).

## 4. 기능 요구사항 (화면별)

> 화면 와이어프레임은 마스터 문서 v3.1 부록 A 참조. 아래는 기능 명세.

### 4.1 공통
- **랜딩**: 두 갈래 진입(사장님/파트너) + 소셜 로그인 3종 + 신뢰 근거 3종. 매니저 버튼 상시.
- **비전 화면(전환 시 1회)**: 사장님/파트너용 각각. "월 100만원" 전면, 면책 하단. 건너뛰기 가능.
- **곁에 매니저(챗봇)**: 전 화면에서 접근. FAQ 14개 응답 + 사람 연결(카톡 단일 채널).

### 4.2 사장님
| 화면 | 핵심 기능 |
|---|---|
| 일 맡기기 | 자유 서술 입력 + 빠른 칩(집출 서비스 포함) + "곁에에 맡기기". 종류 강제선택 없음 |
| 의뢰 현황 | 4단계 진행바(접수·검토/연결·견적/작업중/완료·검수) + 24h 견적 기한 표시 |
| 견적 승인 | 익명 파트너 카드(연락처 계약후 공개) + 곁에 평가 ★ + 총액(매칭비 통합) + 견적서 PDF + 에스크로 안내 + 승인/금액상의 |
| 검수·정산 | 결과물 확인 + 검수완료/수정요청/문제신고 + 거래명세서 PDF + 검수기한 3일(미확인 자동확정) |

### 4.3 파트너
| 화면 | 핵심 기능 |
|---|---|
| 등록 | 분야 칩(집출 포함) + 경력 구간 + 한줄 소개. "작업료 0% 수수료" 강조. 최소 입력 |
| 의뢰 수락 | 곁에 책정 견적 + 작업 범위 명시 + 납기 + 수락/조건상의/거절 |
| 작업 현황 | "작업료 안전 보관 중"(에스크로) + 받을 작업료 + 결과물 업로드 + **공통 5단계 워크플로우 진행 체크리스트(수집→구조화→생성→검증→납품)** |
| 정산·평가 완료 | 정산 확정 표시 + 곁에 검토진 확인 안내 + 선택적 "전할 말" + 정산내역서 |

### 4.4 관리자 (통합 대시보드)
| 영역 | 핵심 기능 |
|---|---|
| 오늘 할 일 요약 | 매칭대기/진행중/정산대기/새문의 카운트 |
| 매칭 대기 | 새 의뢰 + AI 키워드 추천 후보(적합도·별점) → 사람이 직접 선택 |
| 진행 중 | 거래 상태(작업중/검수대기) 관리 + 파트너 워크플로우 5단계 진행 확인 |
| 정산 관리 | 에스크로 상태 + **정산 실행은 사람 버튼만** (돈 관련 안전장치) |
| 문의 큐 | 챗봇이 못 푼 문의 + 에스컬레이션(정부지원→박이사 등) |
| 곁에 검토 입력 | 거래 후 별 5개 평가 + 검토의견(공개) + 사장님 협조도(내부 비공개) |

## 5. 비즈니스 규칙 (구현 시 반드시 반영)

### 5.1 수수료 (7구간 매칭피 — 작업료 기준)
| 작업료 구간 | 매칭피 |
|---|---|
| 3만원 미만 | 거래 불가 |
| 3~10만원 | 정률 20% (최소 1만원) |
| 10~30만원 | 정률 15% |
| 30~50만원 | 정액 5만원 |
| 50~80만원 | 정액 7만원 |
| 80~300만원 | 정률 7% |
| 300만원 초과 | 정률 5% |

### 5.2 금액 표시 규칙
- **사장님 화면**: 총액 하나(작업료+매칭피 통합). 직거래 방어.
- **파트너 화면**: 작업료만 표시(자기 몫). 매칭피 비노출.
- **법인 서류(견적서·거래명세서)**: 작업료·매칭비 분리 표기 (회계·세무).
- 모든 금액 VAT 별도.

### 5.3 에스크로 + 곁에 책임 적립금 (핵심 안전장치)
- 사장님이 [작업료+매칭피] 선입금 → 곁에 보관 → 검수 완료 후 파트너에 작업료 전달.
- **정산 실행은 반드시 관리자(사람) 버튼으로만.** 자동/챗봇 금지.
- 파트너 연락처는 계약(deal 성사) 전까지 사장님에게 비공개.
- **곁에 책임 적립금(신규)**: 매칭피의 일부(기본 10%, 설정 가능)를 곁에 책임 적립금으로 적립. 파트너 귀책 사고(결과물 부실·미이행 등) 시 사장님 환불·재매칭 비용에 충당하는 2차 안전망.
  - ⚠️ **외부 보험사 보증이 아니다.** 곁에가 자기 매칭피 수입에서 떼어 **별도 계좌에 분리 보관**하는 내부 적립금. 고객에게 '보험 보증'으로 오인시키지 않고 '곁에 책임 적립금'으로 정직하게 표기.
  - ⚠️ **에스크로 자금과 분리 필수**: 에스크로 자금(사장님이 낸 작업료+매칭피, 파트너에게 갈 돈)과 책임 적립금(곁에 수입 일부)은 다른 돈. 운영비와도 분리. 섞이면 사고 시 실제 가용 불가.
  - 성장기: 거래·금액 커지면 외부 보증보험(이행보증·신원보증)으로 전환 검토.

### 5.6 법적 지위 (준수 사항)
- 곁에는 '고용·파견'이 아니라 **'건별 용역 중개 + 통신판매중개'** 포지션. 근로관계가 아닌 건별 도급 중개임을 약관·계약에 명시(파견법 리스크 회피).
- ⚠️ 직업안정법상 '직업소개사업' 신고 대상 여부, 파견법 적용 여부는 **서비스 오픈 전 법무 확인 필수**(TASKS T0.6).

### 5.4 평가
- 별점·점수는 **곁에 3인 검토진이** 매긴다 (사용자 아님).
- 표시: "곁에 평가 ★4.8" + 검토 의견 한줄평.
- 사장님 협조도는 내부 비공개.

### 5.5 권한 (DB RLS와 직결)
- 관리자: 전체 읽기 + 상태 변경.
- 챗봇: 읽기 + 문의(inquiry) 기록만. 거래·정산 상태 변경 절대 금지.
- 돈·계약 행위는 사람 버튼만.

## 6. 비기능 요구사항
- **모바일 우선(mobile-first)**: 주 사용자가 바쁜 사장님·시니어. 반응형 웹.
- **단순화 헌법 6원칙**: 한 화면 한 가지 일 / 텍스트 우선 장식 배제 / 고객 단순·관리자 강력 / 막히면 매니저 / 행동으로 안내 / 절제된 신뢰감.
- **접근성**: 시니어 배려 — 큰 글씨, 명확한 버튼, 최소 입력.
- **정직**: 수익 보장 표현 금지("월 100만원"은 목표·조력). 면책 명시.
