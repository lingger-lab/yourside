# 곁에(yourside) — 파트너 워크플로우 기초 설계서

> **문서 버전**: 1.0 (마스터 v3.4 기반)
> **담당**: Brad (Phase 2~3 주도), MVP 수준 1 최소는 김동현 대표 구현
> **목적**: 파트너에게 "앱 50개"가 아니라 "도메인을 AI로 프로세싱화하는 공통 워크플로우"를 제공하기 위한 기초 설계. 곁에 플랫폼과의 연계 전략 포함.

---

## 0. 핵심 원칙

1. **앱 배포 ❌ → 워크플로우 제공 ✅**: 파트너에게 30앱+20스킬을 던지는 것은 불가능·무의미. 일의 흐름을 AI가 관통하게 한다(똑빌더式).
2. **공통 뼈대 + 도메인 설정**: 문서·디자인·개발이 달라도 일의 뼈대는 같다. 뼈대는 공통, 도메인 특성만 갈아끼운다.
3. **AI가 생산, 사람이 책임**: AI가 1~3단계로 빠르게 만들고, 사람이 4단계(검증)에서 책임지고 다듬는다. = 곁에의 정체성·AI 시대 생존 구조.
4. **곁에의 자산이 되어야 한다**: 워크플로우는 파트너 개인 도구가 아니라 곁에 안에서 돌아 데이터·품질·해자가 된다.

---

## 1. 공통 5단계 뼈대

모든 지식노동(문서·디자인·개발 등)에 공통으로 적용되는 흐름. 곁에 거래 7단계와 맞물린다.

```
[1 수집] → [2 구조화] → [3 생성] → [4 검증★] → [5 납품]
```

| 단계 | 하는 일 | AI 역할 | 사람 역할 | 산출물 |
|---|---|---|---|---|
| 1 수집 | 의뢰·자료·요구사항 수집 | 자료에서 요구사항 추출, 빠진 정보 역질문(휴먼인더루프) | 역질문에 답·미결 확정 | 요구사항 정리표 |
| 2 구조화 | 무엇을 어떤 순서·형태로 | 구조 초안 생성(목차·레이아웃·IA) | 구조 승인·수정 | 구조 설계도 |
| 3 생성 | 구조에 내용 채우기 | 초안 대량 생성(도메인 지식+엔터랩스 도구) | 방향 지시·선택 | 초안 v1 |
| **4 검증★** | **책임지고 검토·교정·고도화** | 일관성·오류 점검 보조 | **최종 판단·책임(도메인 전문성)** | 검증된 결과물 |
| 5 납품 | 제출 + 곁에 검수 연결 | 포맷 정리·체크리스트 | 제출, 곁에 3인 검수 | 최종 납품물 |

> **4단계가 핵심**: AI가 빠르게 만들고 사람이 책임지고 다듬는 구조. 마스터 §8(리스크)의 결론 "AI가 생산, 사람이 판단·책임"을 작업 단위로 구현. AI가 강해질수록 4단계(사람의 검증·책임)의 가치가 오른다.

---

## 2. 도메인 설정 5요소 (갈아끼우는 부분)

뼈대는 같고, 각 단계의 *내용물*만 도메인마다 다르다. 새 도메인 추가 = 아래 5개 설정을 채우는 것(뼈대는 안 건드림).

| # | 설정 요소 | 정의 | 예: 정부과제 문서 |
|---|---|---|---|
| 1 | 수집 체크리스트 | 무엇을 받아야 하나 | 공고문·기업정보·심사기준 |
| 2 | 구조 템플릿 | 어떤 뼈대로 | PSST 목차·심사항목 매핑 |
| 3 | 생성 도구·프롬프트 | 어떤 엔터랩스 앱·스킬 | 사업계획서 생성 앱 |
| 4 | **검증 기준** | 무엇을 봐야 통과인가(도메인 전문성 핵심) | 심사 통과 관점 점검 |
| 5 | 납품 포맷 | 어떤 형태로 | 정부 양식 변환 |

### 도메인 적용 예시 (3종)
| 단계 | 문서작성(정부과제) | 디자인 | 웹개발 |
|---|---|---|---|
| 수집 | 공고문·기업정보·심사기준 | 브랜드·레퍼런스·용도 | 요구사항·사용자·기능 |
| 구조화 | PSST 목차·심사 매핑 | 레이아웃·무드보드 | IA·기능명세·화면흐름 |
| 생성 | 항목별 초안 | 시안 생성 | 코드·컴포넌트 |
| 검증 | 심사 통과 관점 | 브랜드 정합·완성도 | 동작·품질 테스트 |
| 납품 | 정부 양식 변환 | 파일 포맷·가이드 | 배포·문서화 |

## 3. 곁에 플랫폼 연계 전략 ★

### 3.1 왜 곁에 안에서 돌아야 하나
- **데이터**: 워크플로우가 곁에 안에서 돌면 작업 데이터가 곁에 자산이 됨(AI 매칭 학습 교재). 밖에서 돌면 곁에는 아무것도 못 얻음.
- **품질**: 곁에가 워크플로우를 제공하면 파트너별 들쭉날쭉한 품질이 표준화됨 → "곁에 검증"의 근거.
- **락인·해자**: 파트너가 곁에 워크플로우에 익숙해지면 떠나기 어려움. 도구가 해자가 됨(마스터 0.5.4 차별점 실현).

### 3.2 균형 원칙 (과설계 방지)
곁에는 **"워크플로우의 뼈대와 연결점"**을 제공하고, 실제 작업 도구는 유연하게. 곁에가 모든 도메인 도구를 다 만들지 않는다. 5단계 흐름을 곁에가 잡아주고, 각 단계 도구는 엔터랩스 자산 또는 파트너 선택.

### 3.3 연계 3수준 (얕음 → 깊음, 단계적)
| 수준 | 내용 | 타이밍 | 무게 |
|---|---|---|---|
| **수준 1 — 연결** | 거래에 5단계 워크플로우 가이드가 붙고, 진행을 추적 | **MVP(최소 조각)** | 가벼움 |
| 수준 2 — 도구 임베드 | 각 단계에 엔터랩스 앱·스킬이 곁에 안에서 실행 | Phase 2 | 중간 |
| 수준 3 — 학습 루프 | 작업 데이터로 AI 매칭·워크플로우 자동 개선 | Phase 3 | 깊음 |

> 순서 근거: 데이터 없으면 수준 3 불가, 도구 없으면 수준 2 불가 → 자연히 1→2→3. 수준 1은 코드가 거의 안 들고(가이드+체크리스트), 수준 2는 도구 임베드, 수준 3은 데이터 축적 후.

### 3.4 곁에 화면 연결점
| 곁에 화면 | 워크플로우 연결 | 수준 |
|---|---|---|
| 파트너 등록 | 도메인 선택 → 워크플로우 매칭 | 1 |
| 파트너 의뢰 수락 | 수락 시 5단계 가이드 제공 | 1 |
| 파트너 작업 현황 | 5단계 진행 추적(체크리스트) | 1→2 |
| (신규) 워크플로우 작업대 | 각 단계 AI 도구 실행 공간 | 2 |
| 관리자 대시보드 | 워크플로우 데이터 → 매칭·품질 인사이트 | 3 |

> **1차 연결점 = 기존 "작업 현황" 화면의 진화**. 완전히 새 화면이 아니라 단순 업로드 → 5단계 체크리스트로 고도화. 자연스럽고 MVP를 키우지 않음.

---

## 4. MVP에 들어갈 "수준 1 최소" (확정 범위)

박이사·김동현 대표에게 "워크플로우가 거래에 붙어 작동한다"를 보여주는 개념 증명(PoC). MVP를 비대하게 만들지 않는 최소 조각만.

### 포함 ✅
- **공통 5단계 워크플로우 1개**(도메인 구분 없는 공통 뼈대 v0)
- **작업현황 화면에 5단계 진행 체크리스트**(기존 단순 업로드 → 진화)
- **단계별 완료 체크 + 관리자 대시보드에서 진행 확인**

### 제외 ❌ (이후)
- 도메인별 워크플로우(설정 5요소) → Phase 2
- 단계별 AI 도구 임베드(수준 2) → Phase 2
- 데이터 학습 루프(수준 3) → Phase 3

> MVP는 "공통 5단계 체크리스트가 작업현황에 붙고, 파트너가 체크하며 진행하고, 관리자가 본다"까지. 이것으로 개념은 충분히 증명되고, 도구 임베드·도메인화는 "다음에 이게 들어갑니다"로 설명.

---

## 5. 엔터랩스 자산 매핑 (Phase 2~, 참고)

30앱+20스킬을 던지는 게 아니라, 뼈대의 각 단계에 *맞는 것만* 곁에가 끼워준다. 파트너는 50개를 배울 필요 없이 자기 워크플로우에 들어온 몇 개만 사용.

| 단계 | 활용 자산(예) |
|---|---|
| 수집·구조화 | RAG 앱(과거 자료), 기획 앱 |
| 생성 | 문서 생성·콘텐츠 앱·도메인별 앱 |
| 검증 | 진단·점검 스킬 |
| 즉석 스킬 생성 | 특정 도메인 검증 기준(4번)을 미니앱으로 |

## 7. 한 줄 요약

> 파트너 워크플로우의 본질은 "AI가 1~3단계로 빠르게 만들고 사람이 4단계에서 책임지고 다듬는" 공통 뼈대다. 도메인이 바뀌어도 뼈대는 같고 5개 설정값만 갈아끼운다. 이 워크플로우는 곁에 안에서 돌아 데이터·품질·해자가 되어야 하며, MVP엔 그 최소 조각(5단계 체크리스트+추적)만 넣어 개념을 증명하고 단계적으로 깊여간다.
