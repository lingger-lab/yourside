# 곁에(yourside) — 아키텍처 문서

> **문서 버전**: 1.0 (마스터 v3.1 기반)
> **선행 문서**: `01_PRD.md`
> **목적**: 시스템을 어떤 구조·기술로 구성하고 어떻게 배포하는지 정의한다.

---

## 1. 아키텍처 원칙

1. **직접 만들 것을 최소화** — 인증·DB·저장은 Supabase에 위임. 곁에 비즈니스 로직에 집중.
2. **MVP는 단순하게** — Next.js 풀스택(별도 백엔드 서버 없음). 복잡 로직(AI 매칭 등)이 커지면 그때 분리.
3. **외부 모듈 느슨한 결합** — RAG·챗봇은 API로 연동, 내부 구현 비의존.
4. **갇히지 않기** — Supabase는 표준 PostgreSQL. 확장 시 Cloud Run + Cloud SQL로 이전 가능.

## 3. 기술 스택

| 레이어 | 기술 | 역할 |
|---|---|---|
| 프론트엔드 | **Next.js 14+ (App Router)** + TypeScript | 화면·라우팅 |
| 스타일 | **Tailwind CSS** | 디자인(딥블루·네이비, 단순화 헌법) |
| 백엔드 | **Next.js API Routes** | 비즈니스 로직(서버리스) |
| DB | **Supabase (PostgreSQL)** | 데이터 |
| 인증 | **Supabase Auth** | 소셜 로그인(구글·카카오·네이버) |
| 파일 저장 | **Supabase Storage** | 결과물·견적서 파일 |
| 권한 | **Supabase RLS** | 행 단위 접근 제어 |
| 결제 | **토스페이먼츠 결제위젯** | 에스크로 |
| 배포(앱) | **Vercel** | git push 자동 배포 |
| 도메인 | yourside.cloud | Vercel DNS 연결 |
| RAG·챗봇 | **공용 모듈(외부, API 연동)** | MVP 2단계 |
| 문서 PDF | 서버리스 PDF 생성 | 견적서·거래명세서 |

## 4. 배포 구성

```
[개발] Claude Code로 코드 작성
   │
   ▼ git push (GitHub)
   ├──────────────────────────┐
   ▼                          ▼
[Vercel]                  [Supabase]
- Next.js 자동 빌드·배포     - 프로젝트 생성(배포 불필요)
- yourside.cloud 연결        - DB·Auth·Storage 클라우드 상주
- HTTPS 자동                 - 마이그레이션은 SQL로 관리
   │
   ▼ 외부 연동
[토스페이먼츠] 결제 API
[카카오 채널] 사람 연결
[공용 RAG·챗봇 모듈] (2단계)
```

- **서버 운영 0**: Vercel·Supabase가 인프라 관리. 3인 팀은 코드·비즈니스에 집중.
- **자동 배포**: GitHub 푸시 → Vercel 자동 빌드. 환경변수는 Vercel·Supabase 대시보드에서 관리.
- **비용**: 초기 무료 티어. 트래픽 증가 시 각 월 $20~25. 토스는 거래 건당 수수료.

## 6. 핵심 모듈 책임

### 6.1 견적 계산 (`lib/pricing.ts`)
- 입력: 작업료(원)
- 출력: 매칭피(7구간 규칙), 총액, VAT
- 3만 미만은 거래 불가 반환. (PRD 5.1)

### 6.2 키워드 매칭 (`lib/matching.ts`)
- 입력: 의뢰 텍스트(title+detail)
- 처리: 파트너 분야·소개 텍스트와 키워드 일치도 계산(단순 점수)
- 출력: 적합도 순 파트너 후보 리스트 (관리자 보조용, 고객 비노출)
- **임베딩·벡터 검색 사용 안 함** (MVP)

### 6.3 챗봇 (`lib/chat/`)
- **MVP 1단계**: FAQ 14개 정적 매칭 응답 + 에스컬레이션 분기.
- **MVP 2단계**: 공용 RAG·챗봇 모듈 API 호출로 교체. 인터페이스는 TRD 참조.
- 챗봇은 DB 읽기 + inquiry 기록만. 상태 변경 금지.

### 6.4 결제 (`api/payments/`)
- 토스페이먼츠 결제위젯으로 에스크로 선입금.
- webhook으로 입금 확인 → settlement 상태 갱신.
- 정산 실행(파트너 전달)은 관리자 버튼 트리거.
- **곁에 책임 적립금**: 정산 시 매칭피의 일부(기본 10%)를 원장에 적립. 사고 시 환불·재매칭 비용 충당(2차 안전망). 외부 보험 아닌 내부 적립금(별도 계좌, 에스크로 자금·운영비와 분리). append-only 원장.

## 7. 데이터 흐름 예시 (의뢰→정산)

```
1. 사장님 일 맡기기 → POST /api/requests → request 생성(status: open)
2. 관리자 대시보드 → GET /api/matching?request_id → 키워드 추천 후보
3. 관리자 파트너 선택 → POST /api/matching → matching 생성(status: accepted 대기)
4. 파트너 수락 → matching.status = accepted → deal 생성(견적 계산)
5. 사장님 승인+결제 → 토스 결제위젯 → webhook → settlement(escrow: deposited)
6. 파트너 결과물 제출 → Supabase Storage 업로드 → deal.status = done 대기
7. 사장님 검수 완료 → deal.status = done
8. 관리자 정산 실행 → settlement(escrow: released) → 파트너 전달
9. 관리자 곁에 검토 입력 → review 생성(곁에 평가 별점·의견)
```

## 8. 확장 경로 (MVP 이후, 참고)
- **트래픽 증가**: Vercel → Cloud Run, Supabase → Cloud SQL 이전(PostgreSQL 호환).
- **AI 매칭 고도화**: 키워드 → 임베딩·벡터 검색(손매칭 데이터 학습 후).
- **챗봇**: 공용 RAG 모듈 정식 연동 → 거래 데이터 적재.
- **도구 구독**: 결제 시스템 확장(기본 무료 유지, 고급 선택 구독).
