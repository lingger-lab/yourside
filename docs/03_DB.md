# 곁에(yourside) — DB 설계 문서

> **문서 버전**: 1.0 (마스터 v3.1 기반)
> **선행 문서**: `01_PRD.md`, `02_ARCHITECTURE.md`
> **DB**: Supabase (PostgreSQL)
> **목적**: 데이터 모델·테이블·관계·권한을 정의한다. 전체 실행 SQL은 §7.

---

## 1. 설계 원칙

1. **소셜 로그인 + 최소 정보**: 가입 필수는 이메일·연락처만. 나머지는 nullable(점진 수집).
2. **상태는 enum**: 거래·정산 상태를 enum으로 강제 → 잘못된 값 원천 차단.
3. **상태 변경은 사람만**: 돈·계약 상태는 관리자 버튼으로만. 챗봇은 읽기 + inquiry 기록만.
4. **단계적 구현**: 극초기 4테이블(client·partner·request·deal)부터. matching·settlement은 거래량 쌓이면 승격.

---

## 2. 테이블 개요 (8개)

| 테이블 | 역할 | 거래 단계 | MVP 우선순위 |
|---|---|---|---|
| `client` | 사장님(수요자) | — | 1차 |
| `partner` | 파트너(공급자) | — | 1차 |
| `request` | 의뢰 | 1단계 | 1차 |
| `deal` | 거래(작업료·매칭피·총액) | 3~5단계 | 1차 |
| `matching` | 매칭(3인 선별) | 2단계 | 2차(수기 가능) |
| `settlement` | 정산(에스크로) | 4·6단계 | 2차(수기 가능) |
| `review` | 평가(곁에 검토진) | 7단계 | 2차 |
| `inquiry` | 문의(챗봇 큐) | — | 2차 |

> 추가: `guarantee_fund_ledger`(곁에 책임 적립금 원장) — settlement의 매칭피 일부 적립·사고 충당을 추적. settlement에 `guarantee_fee`·`refunded_amt` 필드 포함. ⚠️ 외부 보험사 보증이 아닌 곁에 내부 적립금(별도 계좌, 에스크로 자금과 분리). 고객에겐 '곁에 책임 적립금'으로 표기. (마스터 v3.3 §8.4)

---

## 3. ERD (관계)

```
client ──1:N── request ──1:0..1── matching ──1:1── deal ──1:1── settlement
                                      │                 │
                          partner ───┘                 ├──1:N── review
                                                        │
client/partner ──1:N── inquiry                          
```

- `client` 1 : N `request` (사장님 한 명이 여러 의뢰)
- `request` 1 : 0..1 `matching` (의뢰당 매칭 0 또는 1)
- `partner` 1 : N `matching` (파트너가 여러 건에 선별)
- `matching` 1 : 1 `deal` (성사 시 거래 1건)
- `deal` 1 : 1 `settlement`, 1 : N `review`(양방향 평가)
- `client`/`partner` 1 : N `inquiry`

---

## 4. 소셜 로그인 연동

- Supabase Auth가 `auth.users`에 소셜 가입자 저장(이메일·provider 자동).
- `client`/`partner`에 `auth_user_id`(FK→auth.users), `provider`, `email` 보관.
- **이메일 복사는 트리거 아닌 앱 로그인 로직에서 처리** (auth.users → client/partner).
- 가입 직후엔 이메일만 있고, 나머지(상호·분야 등)는 첫 의뢰/등록 시 채움.

---

## 5. 집출 서비스 반영

- 별도 테이블 불필요. `request.req_type` / `partner.field`에 집출 분야가 **값으로** 들어감.
- 예: req_type/field = "창업코칭", "AEO", "AI전환진단" 등.
- 분류(req_type)는 **관리자가 입력**(사장님 자유 서술 → 박이사가 분류). 사장님 강제선택 아님.

---

## 6. 권한 규칙 (RLS 설계 기준)

| 주체 | 읽기 | 쓰기 |
|---|---|---|
| 사장님 | 자기 client·request·deal·settlement | request 생성, deal 검수확인, review(전할말) |
| 파트너 | 자기 partner·배정된 matching·deal | partner 프로필, 결과물 업로드 |
| 관리자(3인) | 전체 | 전체 상태 변경(매칭·거래·정산·검토) |
| 챗봇 | 읽기 허용 범위 | **inquiry 생성만** |

**철칙**: 정산 실행·매칭 확정 등 돈·계약은 관리자(service_role)만. 파트너 연락처는 deal 성사 전 비공개.

## 7. 전체 실행 SQL (Supabase)

```sql
-- ============================================================
-- 곁에(yourside) MVP 스키마 v1.1 (마스터 v3.1)
-- Supabase (PostgreSQL). 한 번에 전체 실행 가능.
-- ============================================================

-- updated_at 자동 갱신 함수
create or replace function set_updated_at()
returns trigger as $$
begin new.updated_at = now(); return new; end;
$$ language plpgsql;

-- enum 타입
create type auth_provider   as enum ('google','kakao','naver');
create type client_status   as enum ('active','inactive');
create type partner_status  as enum ('active','waiting','suspended');
create type partner_grade   as enum ('veteran','standard','new');
create type request_status  as enum ('open','matching','dealt','closed');
create type matching_status as enum ('proposed','accepted','rejected');
create type deal_status     as enum ('quoted','working','done');
create type escrow_status   as enum ('pending','deposited','reviewing','released','refunded');
create type review_author   as enum ('client','partner','gyeotae');  -- 곁에 검토진 추가
create type inquiry_status  as enum ('open','ai_answered','human_routed','closed');
create type manager_name    as enum ('park','brad','kim');

-- 1. CLIENT (사장님)
create table client (
  id            uuid primary key default gen_random_uuid(),
  auth_user_id  uuid unique references auth.users(id) on delete cascade,
  provider      auth_provider,
  email         text not null,           -- [가입 필수] 소셜 자동 / 마케팅용
                                          -- ※ 이메일 복사는 앱 로그인 로직에서 auth.users→이 테이블
  contact       text,                     -- [선택] 연락처
  company       text,                     -- [선택] 상호명, 첫 의뢰 시
  ceo_name      text,                     -- [선택]
  region        text,                     -- [선택] 부울경 세부지역
  industry      text,                     -- [선택]
  status        client_status not null default 'active',
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

-- 2. PARTNER (파트너)
create table partner (
  id            uuid primary key default gen_random_uuid(),
  auth_user_id  uuid unique references auth.users(id) on delete cascade,
  provider      auth_provider,
  email         text not null,           -- [가입 필수]
  contact       text,                     -- [선택] 계약 전 비공개
  name          text,                     -- [선택]
  field         text,                     -- [선택] 분야(집출 서비스 포함: 창업코칭/AEO/AI전환진단 등)
  career_yrs    int,                      -- [선택] 경력 구간
  grade         partner_grade not null default 'standard',
  status        partner_status not null default 'active',  -- suspended=직거래 제재
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

-- 3. REQUEST (의뢰) - 1단계
create table request (
  id          uuid primary key default gen_random_uuid(),
  client_id   uuid not null references client(id) on delete cascade,
  title       text not null,
  detail      text not null,            -- 자유 서술
  req_type    text,                     -- 관리자 분류(집출 포함). 사장님 강제선택 아님
  scope       text,                     -- 작업 범위(분쟁 방지, 수락 시 명시)
  budget_hope int,                      -- 희망예산(선택)
  status      request_status not null default 'open',
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);

-- 4. DEAL (거래) - 3~5단계
create table deal (
  id          uuid primary key default gen_random_uuid(),
  matching_id uuid unique references matching(id),  -- 수기 단계엔 nullable 허용
  request_id  uuid references request(id),          -- 수기 매칭 대비 직접 참조도 보관
  partner_id  uuid references partner(id),
  work_fee    int not null,             -- 작업료(파트너 몫)
  match_fee   int not null,             -- 매칭피(7구간 계산)
  total_pay   int not null,             -- 작업료+매칭피 (VAT 별도)
  scope       text,                     -- 확정 작업 범위
  due_date    date,                     -- 납기
  status      deal_status not null default 'quoted',
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);

-- 5. MATCHING (매칭) - 2단계 (2차, 수기 가능)
create table matching (
  id          uuid primary key default gen_random_uuid(),
  request_id  uuid not null references request(id) on delete cascade,
  partner_id  uuid not null references partner(id),
  manager     manager_name,             -- 연결한 검토진
  status      matching_status not null default 'proposed',
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);

-- 6. SETTLEMENT (정산) - 4·6단계 (2차)
create table settlement (
  id            uuid primary key default gen_random_uuid(),
  deal_id       uuid not null unique references deal(id) on delete cascade,
  escrow_status escrow_status not null default 'pending',
  payment_key   text,                   -- 토스페이먼츠 결제 키
  guarantee_fee int not null default 0, -- 곁에 책임 적립금 적립액(매칭피의 일부, 예: 10%). 외부보험 아닌 내부적립
  refunded_amt  int not null default 0, -- 환불액(사고·분쟁 시, 책임 적립금에서 충당)
  refund_reason text,                   -- 환불 사유
  deposited_at  timestamptz,            -- 선입금 시각
  released_at   timestamptz,            -- 파트너 전달 시각(정산 실행=사람)
  refunded_at   timestamptz,            -- 환불 시각
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

-- 6-b. RESPONSIBILITY_FUND_LEDGER (곁에 책임 적립금 원장) - 적립·사용 추적
-- ※ 외부 보험사 아닌 내부 적립금. 별도 계좌 보관. 에스크로 자금과 분리.
create table guarantee_fund_ledger (
  id            uuid primary key default gen_random_uuid(),
  settlement_id uuid references settlement(id),
  entry_type    text not null,          -- 'accrue'(적립) | 'payout'(사고 충당)
  amount        int not null,           -- 적립/출금액(원)
  balance_after int,                    -- 거래 후 잔액(스냅샷)
  note          text,
  created_at    timestamptz not null default now()
);

-- 7. REVIEW (평가) - 7단계 (곁에 검토진이 입력)
create table review (
  id          uuid primary key default gen_random_uuid(),
  deal_id     uuid not null references deal(id) on delete cascade,
  author_type review_author not null,   -- 'gyeotae'=곁에 검토진 별점(공개), 'client'/'partner'=선택적 전할말
  rating      int check (rating between 1 and 5),  -- 곁에 별점
  comment     text,                     -- 공개 검토 의견
  internal_note text,                   -- 사장님 협조도(내부 비공개)
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);

-- 8. INQUIRY (문의) - 챗봇 큐
create table inquiry (
  id          uuid primary key default gen_random_uuid(),
  author_id   uuid,                     -- client/partner id
  author_type review_author,            -- client/partner
  category    text,                     -- FAQ/정부지원/진단/기술/분쟁
  content     text not null,
  status      inquiry_status not null default 'open',
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);

-- updated_at 트리거 (8테이블)
do $$
declare t text;
begin
  foreach t in array array['client','partner','request','deal','matching','settlement','review','inquiry']
  loop
    execute format('create trigger trg_%I_updated before update on %I
      for each row execute function set_updated_at();', t, t);
  end loop;
end $$;

-- 인덱스
create index idx_request_client    on request(client_id);
create index idx_request_status    on request(status);
create index idx_deal_request      on deal(request_id);
create index idx_deal_partner      on deal(partner_id);
create index idx_matching_request  on matching(request_id);
create index idx_matching_partner  on matching(partner_id);
create index idx_settlement_deal   on settlement(deal_id);
create index idx_review_deal       on review(deal_id);
create index idx_inquiry_status    on inquiry(status);

-- RLS 활성화
alter table client     enable row level security;
alter table partner    enable row level security;
alter table request    enable row level security;
alter table deal       enable row level security;
alter table matching   enable row level security;
alter table settlement enable row level security;
alter table review     enable row level security;
alter table inquiry    enable row level security;

-- RLS 정책 (핵심 골격 — auth.uid() 매핑은 구현 시 조정)
-- 관리자(service_role)는 RLS 우회(기본). 정산·매칭 등 상태 변경은 service_role로만 수행.
-- 사장님: 자기 의뢰 조회/생성
create policy client_select_own_request on request for select
  using (client_id in (select id from client where auth_user_id = auth.uid()));
create policy client_insert_request on request for insert
  with check (client_id in (select id from client where auth_user_id = auth.uid()));
-- 파트너: 배정된 매칭 조회
create policy partner_select_matching on matching for select
  using (partner_id in (select id from partner where auth_user_id = auth.uid()));
-- INQUIRY: 본인 문의 생성(챗봇 경로)
create policy inquiry_insert on inquiry for insert
  with check (author_id is not null);
-- (그 외 정책은 구현 단계에서 화면별 접근에 맞춰 추가)
```

## 8. 주의 (구현 메모)
- `deal.matching_id`를 nullable로 둬, 극초기 수기 매칭(matching 테이블 미사용) 시에도 deal 생성 가능.
- `review.author_type='gyeotae'`가 공개 별점(곁에 평가). client/partner는 선택적 전할말·내부기록.
- enum 확장은 `alter type ... add value`로.
- 결제 키(`settlement.payment_key`)는 토스 webhook 처리에 사용.
